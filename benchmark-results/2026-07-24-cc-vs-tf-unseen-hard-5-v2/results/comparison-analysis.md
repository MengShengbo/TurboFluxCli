# FastContext vs. Claude Code：5 道陌生高难任务对比分析

## 结论

本轮两套系统均在 5/5 任务中把唯一 Gold 实现文件排在第一位，Recall@1、Recall@10、MRR、MAP 与 nDCG@10 全部为 1.000。就这五道题而言，FastContext 已达到 Claude Code 同一质量档位，但没有形成检索质量优势；样本量也不足以支持“全面超越”的统计结论。

FastContext 的主要收益是更少的模型请求和输入 Token，以及更稳定的最坏延迟；主要问题是中位延迟仍高 23.5 秒，同时产生了约 6.5 倍的本地工具操作。当前最准确的判断是：**质量打平，模型侧成本更低，尾延迟更稳，但常态速度与本地检索效率仍落后。**

## 实验设置

- 任务：从真实 issue 和修复前仓库快照中定位需要修改的实现文件。
- 数据：5 个历史测试未使用的高难任务，来自 5 个仓库；每题只有 1 个 Gold 实现文件。
- 系统：FastContext 与 Claude Code 2.1.177，只开放只读检索能力。
- 模型：`gpt-5.5`，两套系统均关闭原生推理。
- 运行：每题每系统 1 次，单次超时 300 秒；系统顺序按任务轮换。
- 成功：10/10，无超时；Claude Code 出现 1 次可恢复 API 重试，FastContext 为 0。

## 逐题结果

| 任务 | FastContext | Claude Code | 更快者 | TF / CC 输入 Token | TF / CC 工具调用 |
|---|---:|---:|---|---:|---:|
| `psf__requests-2931` | 86.9s | 43.5s | Claude Code | 22,511 / 37,270 | 59 / 6 |
| `yt-dlp__yt-dlp-10390` | 111.3s | 149.7s | FastContext | 23,903 / 20,795 | 56 / 6 |
| `mui__material-ui-36353` | 90.1s | 58.5s | Claude Code | 8,335 / 21,634 | 55 / 12 |
| `sphinx-doc__sphinx-8721` | 71.0s | 83.5s | FastContext | 16,931 / 19,045 | 58 / 6 |
| `pydata__xarray-4695` | 81.1s | 63.4s | Claude Code | 22,594 / 121,350 | 60 / 14 |

FastContext 赢下 2/5 道速度对比，Claude Code 赢下 3/5。FastContext 在 `xarray` 上只用了 Claude Code 18.6% 的输入 Token，同时保持第一名命中；这说明模型上下文控制确实有效。相反，`requests` 在质量相同的情况下耗时正好约为 Claude Code 的两倍，是最需要拆解的常态延迟样本。

## 日志归因

1. **模型调用并不是数量瓶颈。** FastContext 平均只有 3.8 次 API 请求，低于 Claude Code 的 6.6 次；继续简单压缩请求轮数的收益有限。
2. **本地候选生成仍然过宽。** FastContext 每题执行 55 至 60 次本地操作，其中 39 至 44 次是搜索，最终读取 12 至 18 个文件。虽然这些操作不是等价的 LLM 工具往返，但它们显示前沿收敛仍不够快。
3. **二轮检索不是唯一慢因。** MUI 只进行 1 轮检索仍耗时 90.1 秒；其余四题为 2 轮。延迟不能只归因于反馈轮，单轮 Planner、候选执行和最终 Judge 都需要更细粒度计时。
4. **输出协议偏重。** FastContext 输出 Token 是 Claude Code 的 2.74 倍，原因包括排序理由、角色、证据区间及 compact code map。它提升了交付可用性，但也增加生成尾部延迟。
5. **尾延迟控制有效。** FastContext 最慢任务为 111.3 秒，Claude Code 为 149.7 秒；本轮没有 API 重试，说明请求数量较少和受控轮次对最坏情况有明显价值。

## 当前工程判断

FastContext 已经解决了此前“高难题候选生成失焦导致漏掉 Gold 文件”的核心质量问题：五道陌生高难题全部首位命中，且没有证据门控拒绝或耗尽回合。LLM 主导查询规划、本地并发执行、读后验证和有限反馈这一组合，在本轮真实任务上是成立的。

但它还不是更快的检索系统。FastContext 当前更像“模型成本显著更低、尾部可控、结果证据更丰富”的工程化检索器。下一阶段不应再扩充检索分支，而应集中解决以下三个可测问题：

1. 给 Planner、首批搜索、文件读取、反馈 Planner、Judge、结果序列化分别记录墙钟时间和首/末 Token 时间，修复当前 FastContext `apiDurationMs` 为空的仪表缺陷。
2. 对本地搜索做边际信息增益停止：当 owner 候选已被读证据确认且后续搜索只增加低置信度 supporting 文件时，立即终止搜索前沿。
3. 将最终输出拆成固定短 code map 与按需展开证据，减少默认输出 Token，同时保留主 Agent 可继续验证的文件和行区间。

## 可信边界

- 只有 5 道题且每题只运行 1 次；质量指标饱和后，配对置换检验 `p=1.0`，无法证明系统间存在质量差异。
- 每题只有一个 Gold 实现文件，尚未检验多文件修改、跨语言调用链和间接配置 owner 的完整覆盖能力。
- 延迟受 API 服务状态影响；Claude Code 的 `yt-dlp` 任务发生一次重试，因此最坏延迟优势需要更多重复实验确认。
- FastContext 的 `apiDurationMs` 未记录，当前只能对请求数、工具数和总墙钟时间做归因，不能严谨拆出网络、模型与本地执行占比。

本轮结果可以支持“FastContext 在陌生高难定位任务上与 Claude Code 达到相同命中质量，并显著降低模型输入成本”；不能支持“FastContext 已全面超越 Claude Code”。
